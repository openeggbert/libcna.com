C++ API differences from C# XNA

While CNA mirrors XNA's API surface, there are necessary differences due to the C++ language:

  • No garbage collection: Resources are managed with RAII and std::unique_ptr / std::shared_ptr. Objects must be explicitly managed by the caller.
  • No properties: C# properties become getter/setter methods (e.g. getGraphicsDeviceProperty() instead of the C# GraphicsDevice property).
  • No events: C# event delegates are replaced with virtual method overrides or callbacks where applicable.
  • Namespace syntax: Microsoft::Xna::Framework with :: instead of ..
  • No extension methods, LINQ, or generics as they exist in C#.

Known gaps and limitations

  • XACT audio (AudioEngine, SoundBank, WaveBank, Cue) — stub only, ~0% functional.
  • XNB content pipeline — not supported; CNA uses JSON descriptors instead of XNB binary assets.
  • Touch input — API declared, SDL3 backend not fully wired; needs hardware validation on real touch devices.
  • GamerServices — stub (Guide, GamerServicesComponent, Gamer API); no live Xbox Live services.
  • FillMode::WireFrame — Vulkan backend only; not available on EasyGL.
  • macOS — not an official target (SDL3 supports it but is untested in CNA).
  • BGFX backend — ~82%, missing EnvironmentMapEffect, ShaderEffect, MSAA, and GetBackBufferData.

Intentionally excluded from CNA

  • Xbox Live / GamerServices online features (network multiplayer, achievements, leaderboards) — platform-specific to Xbox; not applicable to a portable C++ library.
  • XACT runtime — will remain a stub unless contributed by the community.
  • XNB binary content pipeline — replaced by JSON descriptors; XNB import is not planned.
  • Windows Phone sensor emulation — the Accelerometer class works where SDL3 supports it; Phone-specific emulation is excluded.
  • Direct3D-specific APIs — GraphicsAdapter enumeration returns a stub on non-Windows platforms.
